# Jot — Implementation Plan

Derived from the PRD in [`../PRD/MeetingTranscriber_PRD_v2026_05_27.md`](../PRD/MeetingTranscriber_PRD_v2026_05_27.md). The goal is a native macOS menu-bar utility that watches a folder for Audio Hijack recordings, transcribes them via a user-configured OpenAI-compatible API endpoint, and organizes each meeting into its own folder containing the original audio and the transcript.

---

## 0. Guiding constraints (from the PRD)

- **Native SwiftUI only.** No Electron / Tauri / cross-platform wrappers.
- **Menu-bar driven, with a real main window.** No Dock icon. Clicking the menu-bar icon toggles a single main window with a sidebar and three tabs (Transcripts / Audit Log / Settings) — not a dropdown menu, not a preferences-only shell.
- **Transcription only.** No LLM summarization. Output of the pipeline is the raw transcript text written next to the original audio.
- **BYO endpoint.** The transcription endpoint, model string, and API key are user-configured. The client must work against any OpenAI-compatible `/audio/transcriptions` endpoint (Groq, OpenAI, self-hosted Whisper).
- **Audio is out of scope.** Audio Hijack is the recording source of truth. Jot never installs a virtual audio driver and never touches the system's audio routing.
- **Event-driven file watching.** Use `FSEvents` / `DispatchSourceFileSystemObject`. No polling.
- **Main thread never blocks.** All network and disk I/O is `async/await` on `URLSession` or actor-isolated.
- **Secrets stay on-device.** API keys go to Keychain, never to UserDefaults, never to disk in plaintext, never to git.
- **Watch Folder must end empty.** Successful processing moves the source audio into the per-meeting output folder. Failed files stay in Watch Folder until retried or manually deleted (PRD §4.3).

### Notes

- **File names inside each feature folder are illustrative**, not contractual — they reflect the intended decomposition but the final names are decided in the PR that creates them. The *folder names* and *the rule that every feature is self-contained* are contractual.
- **`Shared/` starts almost empty.** Move things in only when a primitive is genuinely reused across two or more features. Premature `Shared/` extraction is a common anti-pattern.
- **Headless features (`Watcher`, `Transcription`, `Pipeline`, `Hotkey`, `Shortcut`) sit alongside UI features** — same `Features/` parent, same rules. They're features because they own a bounded responsibility, not because they have a view.
- **There is intentionally no `Models/` or `ViewModels/` top-level folder.** A view-model belongs in the feature it serves; a model that's only used by one feature lives in that feature.

## 1. Project scaffolding (Phase 0)

**Goal:** A buildable empty SwiftUI app with the menu-bar icon, the main window shell (sidebar + 3 empty tabs), and the `Core/Shared/Features` skeleton in place. **Two installable variants from day one** — `Jot` (Release) and `Jot Dev` (Debug) — per [`development-lifecycle.md`](./development-lifecycle.md) §2.

1. Create Xcode project: macOS → App → SwiftUI lifecycle, Swift 5.9+, deployment target macOS 14.0 (Sonoma).
2. **Dual build configurations via xcconfig files.** Create `Jot/Config/Debug.xcconfig` and `Jot/Config/Release.xcconfig`, set them on the project's Debug/Release configurations. Each defines `PRODUCT_BUNDLE_IDENTIFIER`, `PRODUCT_NAME`, `MARKETING_VERSION`, `CURRENT_PROJECT_VERSION`, and `ASSETCATALOG_COMPILER_APPICON_NAME`:
   - Debug → bundle `com.toshonivc.jot.dev`, name `Jot Dev`, app icon `AppIcon-Dev` (tinted variant).
   - Release → bundle `com.toshonivc.jot`, name `Jot`, app icon `AppIcon`.
3. In `Info.plist` set `LSUIElement = YES` so no Dock icon appears.
4. App entrypoint declares two scenes — a `MenuBarExtra` for the icon and a `Window` for the main window:
   ```swift
   @main
   struct JotApp: App {
       @State private var menuBar = MenuBarController()
       var body: some Scene {
           MenuBarExtra { MenuBarIconLabel(state: menuBar.state) }
               label: { Image(systemName: "waveform") }
           Window("Jot", id: "main") { MainWindow() }
       }
   }
   ```
5. `Core/App/MenuBarController.swift` — observes `PipelineState`; clicking the icon toggles the main window via `openWindow`/`dismissWindow`.
6. `Core/App/MainWindow.swift` — `NavigationSplitView` with sidebar listing **Transcripts**, **Audit Log**, **Settings**. Each tab routes to a placeholder view for now.
7. Create the `Core/`, `Shared/`, `Features/` folders under `Jot/` (empty stubs in each so the tree exists from day one). Add `Core/Updates/` as a placeholder for Sparkle wiring (lands in Phase 8).
8. Add a tinted `AppIcon-Dev` asset to `Assets.xcassets` so Debug builds are visually distinct in the menu bar at a glance.
9. Add the `JotTests` test target; create `JotTests/Features/` empty.

**Deliverable:** Both `Jot.app` (Release archive) and `Jot Dev.app` (Debug `⌘R`) build and run. Run them at the same time — two distinct icons in the menu bar, totally separate state. Clicking either icon toggles its own three-tab main window with placeholder content. No Dock icon for either.

## 2. Settings tab + Keychain (Phase 1)

**Goal:** The Settings tab persists everything the PRD requires and stores the API key in Keychain.

1. **`Core/Settings/AppSettings.swift`** — `@Observable` class, single source of truth. Properties: `apiBaseURL: String`, `modelString: String`, `watchFolderBookmark: Data?`, `outputFolderBookmark: Data?`, `recordingHotkey: KeyCombo?`, `launchOnStartup: Bool`. (`apiKey` is *not* a property — it's accessed via the Keychain wrapper.) Persists via `UserDefaults` except API key.
2. **`Core/Settings/Keychain.swift`** — wraps `SecItemAdd` / `SecItemCopyMatching` / `SecItemDelete`. `kSecAttrService = "com.toshonivc.jot"`, `kSecAttrAccount = "api_key"`. Behind a `KeychainProtocol` so tests can inject an in-memory fake.
3. **`Features/Settings/SettingsView.swift`** — root of the Settings tab; lays out the four section subviews.
4. **`Features/Settings/APIConfigSection.swift`** — `TextField` for Base URL (placeholder `https://api.groq.com/openai/v1/audio/transcriptions`), `TextField` for Model String (placeholder `whisper-large-v3`), `SecureField` for API Key, "Test connection" button that fires a single tiny multipart request and reports OK/failure inline.
5. **`Features/Settings/FoldersSection.swift`** — two "Choose…" buttons using `NSOpenPanel(canChooseDirectories: true)`. Persist chosen URLs as security-scoped bookmarks via `URL.bookmarkData(options: .withSecurityScope)` so access survives relaunch under the sandbox.
6. **`Features/Settings/HotkeySection.swift`** — hotkey recorder field (stores the `KeyCombo`; actual global registration lands in Phase 7).
7. **`Features/Settings/SystemSection.swift`** — "Launch on Startup" toggle and "Quit Jot" button.
8. **Validation.** On save, reject blank Base URL / Model / API Key, and reject identical Watch and Output folders.
9. **Tests** (`JotTests/Features/Settings/`):
   - Unit: `AppSettingsTests` (round-trip persistence with injected UserDefaults), `KeychainTests` (against the in-memory fake), `SettingsValidationTests`.
   - Integration: `SettingsPersistenceIntegrationTests` — write settings, simulate relaunch, verify everything round-trips including security-scoped folder access.

**Deliverable:** Open Settings → enter Base URL + Model + Key + folders + hotkey → quit + relaunch → everything restored. "Test connection" returns OK against a real endpoint when keys are valid.

## 3. Folder watcher (Phase 2)

**Goal:** Detect new audio files (`.mp3`, `.m4a`, `.wav`) the moment Audio Hijack finishes writing them.

1. **`Features/Watcher/FolderWatcher.swift`** — actor wrapping `DispatchSourceFileSystemObject` on an `O_EVTONLY` file descriptor opened against the Watch Folder. React to `.write` / `.extend` events. Exposes `AsyncStream<URL>` of ready files.
2. **`Features/Watcher/FileReadinessDetector.swift`** — debounce state machine. A file is "ready" only after `(size, modificationDate)` is stable for ≥ 2 seconds. Per-file state: `seen → growing → stable → emitted`. Inject a `Clock` so tests are deterministic.
3. **Filter.** Accept only files ending in `.mp3`, `.m4a`, `.wav`. Ignore hidden files (`.DS_Store`, dotfiles) and the in-progress temp extensions Audio Hijack uses (e.g., `.tmp`, `.partial`).
4. **Idempotency ledger.** Persist a JSON file in `~/Library/Application Support/Jot/processed-files.json` mapping `(absolutePath, sha256-prefix) → timestamp`. On startup, skip emit for any file already in the ledger.
5. **Lifecycle.** Subscribe to `AppSettings.watchFolderBookmark`; tear down and restart the watcher when the folder changes.
6. **Tests** (`JotTests/Features/Watcher/`):
   - Unit: `FileReadinessDetectorTests` — feeds synthetic `(size, mtime)` snapshots through the state machine using an injected `Clock`; asserts transitions.
   - Integration: `FolderWatcherIntegrationTests` — real `FileManager`, real tmpdir, real incremental writes; asserts the `AsyncStream` emits within 3 seconds of the write settling and does not emit during growth.

**Deliverable:** Drop an `.mp3` into the watched folder → log line `"ready: <filename>"` appears within ~2s of the write settling. Hidden files and partial writes do not emit.

## 4. Transcription client (Phase 3)

**Goal:** A generic OpenAI-compatible transcription client that posts to whatever endpoint the user configured.

1. **`Features/Transcription/TranscriptionClient.swift`** — actor with `func transcribe(audio: URL, baseURL: URL, model: String, apiKey: String) async throws -> String`. Injectable `URLSession` so tests can swap in a `URLProtocol`-mock-backed session.
2. **`Features/Transcription/TranscriptionRequest.swift`** — composes the `multipart/form-data` body. Fields: `file` (streamed off disk), `model` (string), `response_format=text`. Sets `Authorization: Bearer <apiKey>`.
3. **`Features/Transcription/TranscriptionError.swift`** — typed enum: `invalidEndpoint`, `invalidAPIKey`, `fileTooLarge(limitBytes:)`, `serverError(status: Int, body: String)`, `transientNetwork(underlying: Error)`, `timeout`.
4. **Streaming upload.** Use `URLSession.uploadTask(with:fromFile:)` — never load the whole audio into memory.
5. **Timeouts + retry.** 5-minute request timeout. Retry **once** on `transientNetwork` / `timeout`. No retry on `serverError` with 4xx — these are permanent (bad key, bad model name, bad file).
6. **Tests** (`JotTests/Features/Transcription/`):
   - Unit: `TranscriptionRequestTests` — asserts multipart body shape, headers, Bearer auth, model field. `TranscriptionErrorMappingTests` — asserts each HTTP status maps to the right typed error.
   - Integration: `TranscriptionClientIntegrationTests` — `URLProtocol`-mocked session returns canned 200/401/413/500 responses; asserts client returns the right string or throws the right typed error, and that retry happens exactly once on transient failure.

**Deliverable:** Point the client at any OpenAI-compatible endpoint (Groq, OpenAI, local Whisper) with a 30-second test mp3 → it returns the transcript text. Tests exercise success, all error paths, and retry behavior — no live network.

## 5. File organizer (Phase 4)

**Goal:** Per PRD §4.2, lay down the per-meeting folder structure and move the source audio into it.

1. **`Features/Pipeline/FileOrganizer.swift`** — `func organize(audio sourceURL: URL, transcript: String, outputRoot: URL) async throws -> URL`. Returns the URL of the created meeting folder.
2. **Folder name** = source audio filename minus extension. Example: `2026-05-27_14-24_Client_Call.mp3` → `2026-05-27_14-24_Client_Call/`.
3. **Name collision.** If the destination folder already exists, append `-2`, `-3`, … until free. Never overwrite.
4. **Transcript file.** Written as `<meetingName>.txt` (default) inside the new folder. (Extension is `.txt` per PRD §4.2; an `.md` toggle can come later if requested — out of scope for v1.)
5. **Move (not copy) the source audio** into the meeting folder using `FileManager.moveItem(at:to:)`. After success the Watch Folder is empty of that file.
6. **Atomic-ish rollback.** If the move fails after the transcript was written, delete the transcript and the (just-created) folder so we don't leave half-state behind.
7. **Tests** (`JotTests/Features/Pipeline/`):
   - Unit: `FileOrganizerTests` — pure-logic tests on naming (collision suffix sequence).
   - Integration: `FileOrganizerIntegrationTests` — real tmpdir; verifies (a) success path leaves `outputRoot/<name>/<name>.{mp3,txt}` and source is gone, (b) collision creates `-2` etc., (c) simulated move-failure rolls back cleanly.

**Deliverable:** Given an audio URL + transcript string, produces a meeting folder containing both files, with the original audio removed from its source location.

## 6. Pipeline orchestrator + Audit Log (Phase 5)

**Goal:** Chain Watcher → Transcription → FileOrganizer behind a single actor; surface every event to a persistent Audit Log with per-entry retry.

1. **`Features/Pipeline/PipelineState.swift`** — `enum PipelineState { case idle; case processing(URL); case error(URL, String) }`. `@Observable`, broadcasted to MenuBarController and AuditLog.
2. **`Features/Pipeline/ProcessingPipeline.swift`** — actor that consumes the Watcher's `AsyncStream`. **Serial queue** — one file at a time. For each file: state → `processing`, call `TranscriptionClient`, on success call `FileOrganizer`, on either failure write to AuditLog and leave the file in Watch Folder.
3. **`Features/AuditLog/AuditLogEntry.swift`** — `Codable` struct: `id: UUID`, `timestamp: Date`, `kind: Kind` (`info | success | failure`), `sourcePath: String`, `message: String`, `durationMs: Int?`, `retryable: Bool`.
4. **`Features/AuditLog/AuditLogStore.swift`** — actor; reads/writes `~/Library/Application Support/Jot/audit-log.json`. Bounded to the most recent 1000 entries (older entries dropped on append). API: `append(_:)`, `clear()`, `markRetried(id:)`. Exposes `AsyncStream<[AuditLogEntry]>` for the view.
5. **`Features/AuditLog/AuditLogView.swift`** + `AuditLogRow.swift` + `AuditLogViewModel.swift` — chronological list, "Clear Log" button, per-row "Retry" button on `failure` entries.
6. **Retry semantics.** Tapping Retry re-enqueues `sourcePath` into the pipeline. The file is still in Watch Folder (per PRD §4.3), so this is safe.
7. **Tests** (`JotTests/Features/Pipeline/` and `JotTests/Features/AuditLog/`):
   - Unit: `ProcessingPipelineTests` — mock Watcher/Client/Organizer, assert state transitions and audit entries for each path. `AuditLogStoreTests` — append, bounding at 1000, clear, persistence round-trip.
   - Integration: `PipelineIntegrationTests` — real tmpdir, real Watcher, `URLProtocol`-mocked Transcription, real FileOrganizer. Asserts end-to-end success (file in → meeting folder out, audit log entry written) and end-to-end failure (audit log entry with retryable=true, file remains in Watch Folder).

**Deliverable:** Drop an mp3 in Watch Folder → menu bar icon goes to `processing` → meeting folder appears → audit log shows green success row. Drop with bad API key → icon turns red → audit log shows failure row with Retry button → fix key → click Retry → success.

## 7. Transcripts tab (Phase 6)

**Goal:** Browse the Output Folder as a native macOS file view inside the Transcripts tab.

1. **`Features/Transcripts/TranscriptsView.swift`** — list/grid of meeting folders read from `AppSettings.outputFolder`. Default view; refreshes when `outputFolder` changes or when the Pipeline reports a new success.
2. **`Features/Transcripts/TranscriptsViewModel.swift`** — actor wrapping `FileManager`; lists meeting folders sorted by modification date descending. Filters out hidden files.
3. **`Features/Transcripts/MeetingFolderRow.swift`** — row UI: folder name, date, file count. Double-click expands inline to show contents; double-clicking a file opens it in the default macOS app via `NSWorkspace`.
4. **`Features/Transcripts/FileContextMenu.swift`** — right-click menu: "Reveal in Finder" (`NSWorkspace.activateFileViewerSelecting`), "Delete" (with confirm), "Rename" (in-place text field).
5. **Live refresh.** Subscribe to pipeline success events; refresh listing without user action.
6. **Tests** (`JotTests/Features/Transcripts/`):
   - Unit: `TranscriptsViewModelTests` — inject a fake `FileManager`; asserts listing, sort order, hidden-file filter, refresh behavior.

**Deliverable:** Open the main window → Transcripts tab is the default view → shows all meeting folders sorted newest-first → double-click opens a file → right-click for actions.

## 8. Hotkey + Apple Shortcut + Launch on Startup (Phase 7)

**Goal:** Configurable global hotkey triggers the user's Apple Shortcut (which asks for meeting name and starts Audio Hijack). App optionally auto-starts at login.

1. **`Features/Hotkey/HotkeyRegistrar.swift`** — registers a global hotkey via Carbon `RegisterEventHotKey` (still the cleanest API for a single global combo). Behind a `HotkeyRegistrarProtocol` so tests inject a fake. Re-registers when `AppSettings.recordingHotkey` changes.
2. **`Features/Shortcut/ShortcutInvoker.swift`** — runs the user's Apple Shortcut by spawning `/usr/bin/shortcuts run "Jot Start Recording"` via `Process`. Shortcut name configurable in Settings (default `"Jot Start Recording"`). Injectable `ProcessRunner` for tests.
3. Wire hotkey → `ShortcutInvoker.run()`.
4. **`Core/LoginItem/LoginItemManager.swift`** — wraps `SMAppService.mainApp` (macOS 13+). Toggled by `AppSettings.launchOnStartup`. Surface errors (e.g., user denied in System Settings) via the audit log.
5. **Permissions.** Global hotkeys may prompt for Accessibility / Input Monitoring in System Settings. If `RegisterEventHotKey` fails, show a one-time inline alert in the Settings tab linking to the right pane. README documents the steps.
6. **Tests** (`JotTests/Features/Hotkey/` and `JotTests/Features/Shortcut/`):
   - Unit: `HotkeyRegistrarTests` — fake Carbon layer, asserts register/unregister/re-register on settings change. `ShortcutInvokerTests` — fake `ProcessRunner`, asserts correct argv. `LoginItemManagerTests` — fake `SMAppService` wrapper, asserts enable/disable.
   - Integration: `HotkeyIntegrationTests` — real `RegisterEventHotKey` against a synthetic event; gated to skip in CI if Accessibility permission isn't granted.

**Deliverable:** Configure hotkey → press it from any app → Audio Hijack starts recording. Toggle Launch on Startup → quit → log out / log back in → Jot reappears in the menu bar.

## 9. Icon state machine + sandbox + Sparkle (Phase 8)

**Goal:** Polished menu-bar icon with real visual states, an error inspector, a green-lit App Sandbox, shippable logging, and Sparkle wired so the production app can receive updates.

1. **Menu-bar icon state machine.** `MenuBarController` subscribes to `PipelineState`. Three asset variants in `Assets.xcassets`: monochromatic Idle, animated Processing (rotating arc or pulsing dot), red Error (exclamation glyph).
2. **Error inspector.** Clicking the menu bar in the Error state (or clicking an audit log failure row) opens a small modal: full error message + "Copy details" + "Open Audit Log" button. No silent failures.
3. **App Sandbox + entitlements.** Enable App Sandbox; declare:
   - `com.apple.security.app-sandbox`
   - `com.apple.security.network.client`
   - `com.apple.security.files.user-selected.read-write`
   - `com.apple.security.files.bookmarks.app-scope`
   - (If hotkey registration needs it: prompt the user to grant Accessibility — does not require an entitlement.)
4. **Verify bookmarks under sandbox.** Confirm `URL.startAccessingSecurityScopedResource()` actually works for the chosen Watch and Output folders after a relaunch in a release build.
5. **Logging.** `os.Logger` with subsystem `com.toshonivc.jot`. One category per feature folder (`watcher`, `transcription`, `pipeline`, …). Ship at `info` level. Hidden `⌥`-click on the menu-bar icon toggles `debug` for the current session.
6. **Sparkle SDK integration.** Per [`development-lifecycle.md`](./development-lifecycle.md) §4.2:
   - Add Sparkle via Swift Package Manager: `https://github.com/sparkle-project/Sparkle`.
   - Generate the EdDSA key pair via Sparkle's `generate_keys` tool. Commit the **public** key to `Info.plist` as `SUPublicEDKey`. Store the **private** key as a GitHub Actions secret named `SPARKLE_PRIVATE_KEY` — never in the repo.
   - `Info.plist` keys: `SUFeedURL` = appcast URL (see Phase 9 for hosting), `SUEnableInstallerLauncherService = YES`, `SUEnableAutomaticChecks = YES`, `SUScheduledCheckInterval = 86400` (24h).
   - `Core/Updates/SparkleUpdater.swift` — thin wrapper around `SPUStandardUpdaterController` so the rest of the app talks to a protocol, not the framework directly. Exposes `checkForUpdates()` for a manual button.
   - Wire a "Check for Updates…" button into `Features/Settings/SystemSection.swift`.
   - **Debug builds skip Sparkle entirely** (compiled out via `#if DEBUG` or a Debug-only build setting) — the Dev app is updated by `⌘R`, not by downloading itself.
7. **README setup guide.** Manual install steps: create the Apple Shortcut, configure Audio Hijack output path, grant Accessibility on first hotkey press, where to find logs.

**Deliverable:** Fresh-machine install of the Release build: enter API config + folders + hotkey + autostart, record a meeting, see it transcribed and filed in the Transcripts tab without ever opening Xcode. "Check for Updates" in Settings successfully reaches the appcast feed and reports "you're up to date" (or installs a newer version if one was published).

## 10. Distribution + CI/CD (Phase 9)

**Goal:** A self-serve pipeline that turns a `git tag v0.X.Y` into a signed `Jot.app.zip` attached to a GitHub Release and an updated appcast that the production app sees automatically. End-to-end mechanics in [`development-lifecycle.md`](./development-lifecycle.md) §3–§5.

1. **`.github/workflows/tests.yml`** — triggers on every push and PR. Steps: checkout, set up Xcode (matching the local minimum), `xcodebuild test -scheme Jot -destination 'platform=macOS'`. Required check on `main` (configure branch protection so PRs can't merge red).
2. **`.github/workflows/release.yml`** — triggers on tags matching `v*`. Steps:
   - Checkout, set up Xcode.
   - Resolve `CURRENT_PROJECT_VERSION` from `git rev-list --count main` (monotonic build number — see `development-lifecycle.md` §5.2).
   - (Optional, gated on a `MACOS_SIGNING_IDENTITY` secret being present) Import Developer ID Application certificate from secrets.
   - `xcodebuild archive` with the Release configuration → export the `.app` from the archive.
   - (Optional, gated as above) Submit to Apple Notary Service via `xcrun notarytool` and staple.
   - Zip the app: `ditto -c -k --keepParent Jot.app Jot-${VERSION}.zip`.
   - Run Sparkle's `sign_update` against the zip using `SPARKLE_PRIVATE_KEY` from secrets → produces the EdDSA signature.
   - `gh release create v${VERSION}` with the zip attached.
   - Regenerate `appcast.xml` via Sparkle's `generate_appcast` (scans the releases folder, includes the signature).
   - Commit and push `appcast.xml` back to `main` (use a `[skip ci]` marker in the commit message to avoid re-triggering tests).
3. **`appcast.xml`** — lives at the repo root. Sparkle reads it from `https://raw.githubusercontent.com/ToshoniVC/MeetingTranscriber/main/appcast.xml`. If the repo is private (it is), either (a) flip the appcast hosting to a separate public repo / GitHub Pages site, or (b) embed a fine-grained read-only PAT into `SUFeedURL` — option (a) is cleaner; document the choice in this section once decided.
4. **Code signing tier.** Default to **ad-hoc** (no secrets required; `xcodebuild` handles it automatically when no identity is set). Steps in `release.yml` that require Developer ID are gated on the secret being present, so the same workflow works for both tiers. Upgrade path documented in `development-lifecycle.md` §4.5.
5. **README sections to add** for users (i.e., you, on a fresh Mac):
   - **First install.** Download `Jot-0.X.Y.zip` from the latest GitHub Release → unzip → drag `Jot.app` into `/Applications` → right-click → Open (one-time Gatekeeper override for ad-hoc-signed builds) → grant Accessibility for the global hotkey when prompted.
   - **Updates.** Happen automatically via Sparkle (on launch + every 24h). Manual trigger: Settings → "Check for Updates…".
   - **Manual install of the Apple Shortcut.** Step-by-step on creating the "Jot Start Recording" shortcut that asks for a meeting name and tells Audio Hijack to start.
   - **Audio Hijack template.** The exact filename template Audio Hijack should use so the watcher picks up files correctly (e.g., `{meeting_name}-{date}.mp3`).
6. **First-release smoke check.** After cutting `v0.1.0`, install the zip from the GitHub Release on a fresh user account (or in a VM if available). Confirm Sparkle's "Check for Updates" reaches the appcast and reports "up to date." This is the gate that confirms the whole loop works before M2 begins.

**Deliverable:** `git tag v0.1.0 && git push origin v0.1.0` → 5 minutes later, the GitHub Release exists with the signed zip attached, `appcast.xml` is updated on `main`, and a manual install of that zip can later be updated to `v0.1.1` simply by clicking "Install Update" in the Sparkle dialog.

## 11. Out of scope for v1

- LLM summarization (explicitly excluded by PRD §1).
- Live / streaming transcription — file-based only.
- Multi-language detection UI (let the endpoint handle language; don't expose a picker).
- Speaker diarization.
- Cloud sync of transcripts.
- Bundling the Apple Shortcut — user creates it manually per the README.
- In-app update / auto-update channel.
- Any GUI for the recording side — Audio Hijack and the Apple Shortcut own that.

## Suggested milestone order

| Milestone | Phases | Demo-able outcome |
|-----------|--------|-------------------|
| M1 | 0, 1 | App in menu bar with main-window shell; Settings tab persists everything; API key in Keychain. |
| M2 | 2, 3, 4 | Headless end-to-end: drop file in Watch Folder → meeting folder appears in Output Folder with the transcript next to the moved audio. No UI for the result yet. |
| M3 | 5, 6 | Live main window: Pipeline state drives the menu-bar icon; Audit Log records every run with working Retry; Transcripts tab browses Output Folder live. |
| M4 | 7, 8, 9 | Hotkey + Apple Shortcut + autostart; sandboxed and logged; Sparkle wired into the Release build; CI publishes signed releases on tag; the production install lives in `/Applications` and updates itself. Daily-driverable. |

Each milestone is its own PR. Note that **Phase 9 (Distribution) should actually land as the first PR of M1** — not at the end of M4 — so the lifecycle infrastructure exists from day one and `v0.1.0` can be cut as soon as Phase 1 is done. The table groups it under M4 because that's when the *full* loop (signed → notified → auto-updated → relaunched without losing state) becomes meaningful end-to-end, but the workflow files and `appcast.xml` scaffolding go in early.
